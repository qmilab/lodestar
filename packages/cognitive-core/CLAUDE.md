# @qmilab/lodestar-cognitive-core тАФ CLAUDE.md

This is where the epistemic chain comes alive. The core consumes typed Observations and produces Claims, EvidenceSets, Beliefs, Decisions, and Explanations.

## What lives here

- **Extractors** (`src/extractors/`) тАФ produce Claims from Observations. One extractor per observation schema (or generic LLM-driven extractor for ad-hoc schemas). The opt-in `DocumentationExtractor` (`documentation.source@1`) reads *into* file content тАФ `package.json` deps/version, markdown title/headings/commands, exported function signatures тАФ for the documentation-agent proving ground; it is not a built-in, so consumers register it explicitly.
- **World model** (`src/world-model/`) тАФ typed key-value store of the agent's current picture of the world. Separate from beliefs (beliefs are *about* world state; the world model captures current observed state).
- **Evidence linker** (`src/evidence-linker.ts`) тАФ for a new claim, find supporting and contradicting sources from prior observations and beliefs. `EvidenceLinkerLike` is the interface `CognitiveCore` depends on (so hosts can inject custom linkers); `DocAwareEvidenceLinker` (`src/doc-evidence-linker.ts`) overrides it to tag `documentation.*` observation content as `external_document` and stamp each evidence item with its source file (`independence_group: doc:<path>`), the same stance the MCP proxy takes for tool-result content.

  **Cross-belief join (#157, ADR-0032).** Beyond the new claim's own source observations, the linker joins the claim against prior beliefs in the same scope that share its `structured_predicate.(subject, relation)` тАФ same `object` тЖТ a `supports` item, different `object` тЖТ a `contradicts` item. The `(subject, relation)` key is the **shared** exported `predicateKey` from `@qmilab/lodestar-memory-firewall` (the same one the firewall's contradiction routing uses тАФ single source, no drift), so the linker now takes a `ClaimStore` constructor dependency to walk `belief.claim_id тЖТ ClaimStore.get`. **Quality inheritance:** a cross-belief item inherits the *quality* and *independence_group* of the prior belief's own strongest supporting evidence (read via the held `EvidenceStore`), so a corroborator can never lend more confidence than its own basis тАФ two `external_document` beliefs stay blocked by the auto-observation (Parallax) gate, while a genuinely stronger independent source clears it (a crisp `unverified тЖТ supported` flip). The logic lives in a `protected crossBeliefItems(claim)` on the base linker that **all four** bodies call (the base plus the `Doc`/`MCP`/`Runtime`-aware subclasses, which override `linkForClaim` rather than calling `super` because `EvidenceStore.put` is a strict insert тАФ so the join cannot live in only one body). The linker stays **pure**: it records items on the *new* claim's evidence only; transitioning the *prior* belief to `contradicted`/`superseded` stays reflection's job (epic child B, #154). **Lifecycle gating:** the join skips any prior belief the firewall has invalidated, isolated, or hard-demoted тАФ non-`clean` security (`quarantined`/`suspicious`/`malicious`), `contradicted`/`superseded` truth, `expired` freshness, or a `retrieval_status` that is not `restricted`/`normal` (so `blocked`/`hidden`/`privileged_only` are out) тАФ `isEligibleJoinPeer` is an allowlist on the retrieval axis (fail-closed). This keeps a quarantined, invalidated, or explicitly-suppressed memory from lending support through this side channel (the "quarantine is one-way" invariant holds transitively). `unverified` and `restricted` peers ARE kept (the Parallax corroboration case; `restricted` is the default adopted state). **Companion fix in `core.ts`:** a claim the join nets *negative* (it contradicts a held belief) is refused as a belief AND no longer overwrites the world model тАФ only positive-evidence claims update observed state, so a rejected claim can't propagate its losing value. Both locked by the `evidence-linker-cross-belief-join` probe (in-memory + Postgres).
- **Planner** (`src/planner.ts`) тАФ turns goals into Decisions and Action proposals, respecting ContextPolicy.
- **Reflection** (`src/reflection.ts`) тАФ on-demand or tail-async; rule-based in v0 (contradicted-belief cascade); produces typed `ReflectionProposal`s and applies them via `MemoryFirewall` with `by_authority: "reflection"`. Stores and the firewall are optional inputs so dry-run inspection (CLI `lodestar reflect`) works without rebuilding live state. Design contract: `docs/architecture/reflection-pass.md`.
- **Explanation generator** (`src/explanation.ts`) тАФ produces structured Explanation records for governance events.

## Invariants

1. **No claim without an observation.** Every Claim references at least one Observation. Free-form LLM extractions without a source observation are invalid.
2. **No belief without evidence.** Every Belief points to a Claim, and adoption goes through the Memory Firewall, which requires an EvidenceSet.
3. **No decision without belief dependencies.** When `ContextPolicy.require_evidence_for_decisions = true`, decisions without `belief_dependencies` are rejected before they reach the planner output.
4. **No silent retrieval.** Every retrieval into model context goes through `@qmilab/lodestar-memory-firewall`'s `GatedRetrieval`. The core does not query the belief store directly.
5. **Reflection never auto-commits.** Reflection outputs are always proposals. The Memory Firewall (with appropriate authority) is the only path to commit.
6. **Working memory is ephemeral.** Anything in `working` is gone at session end. If something needs to survive, it must be promoted to episodic or semantic memory through the firewall.

## What does not live here

- Belief storage and lifecycle transitions: see `@qmilab/lodestar-memory-firewall`.
- Tool registry and execution: see `@qmilab/lodestar-action-kernel`.
- Trust ladder and approval surfaces: see `@qmilab/lodestar-policy-kernel`.
- Probes, sentinels, and calibrators: see `@qmilab/lodestar-harness`.
